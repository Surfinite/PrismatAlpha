## 2. Project Overview

**PrismataAI** is a C++ game engine and AI for **Prismata**, a turn-based perfect-information strategy card game (think "Chess meets StarCraft" with deterministic mechanics). The engine simulates game states. The AI uses Alpha-Beta search, UCT/MCTS, and a PartialPlayer phase decomposition system that breaks each turn into Defense, ActionAbility, ActionBuy, and Breach phases.

**Stage of development:** Active research/improvement. The core engine is mature (forked from academic code by David Churchill). The project owner is training neural network evaluation functions via self-play and iteratively improving hand-coded heuristics. Current best model achieves 51.9% win rate against the baseline AI (`OriginalHardestAI`) — the first model to cross 50%.

**Goal of this plan:** Isolate and measure the effect of a single numeric parameter (`frontlinePenalty`) on AI win rate, via controlled A/B tournament testing on AWS spot instances. This is a focused experiment, not a feature build.

**Constraints:**
- Solo developer, cost-conscious (previous AWS bill shock of $805 for 4 days)
- Experiment budget target: ~$3.36 total
- x86-only C++ codebase (32-bit builds)
- All cloud evaluation uses Windows Server EC2 instances running the compiled exe

## 3. Architecture & Tech Stack

**Languages:** C++ (engine + AI), Python (training pipeline), Bash/PowerShell (cloud orchestration)

**Build:** Visual Studio solution, MSBuild, x86 only (Debug/Release/Static Release). Three executables: GUI, Testing (tournament runner), Standalone.

**AI Architecture — PartialPlayer Phase Decomposition:**

```
Turn Execution Pipeline (per-player per-turn):
  1. Defense       → DefenseSolver (assign blockers)
  2. ActionAbility → ACAvoidBreach_ChillSolver (use abilities)
  3. ActionBuy     → BuyComboGreedy* (purchase new units)  ← FRONTLINE PENALTY LIVES HERE
  4. Breach        → BreachGreedyKnapsack (target enemy units)
```

**Search:** `Player_StackAlphaBeta` with configurable `MoveIterator` (PPPortfolio of PartialPlayers). Each search node evaluates candidate moves generated by combinatorial PartialPlayer portfolios. Evaluation: `NeuralNet` (ResNet, 256h/3L, ~2000 evals/sec/core).

**Config-driven composition:** All AI players, iterators, and partial players are defined in `config.txt` as named JSON entries that reference each other by name. A single player definition like `PrismatAI_AB` references a `MoveIterator` which references `BuyComboGreedy*` entries which reference `BuyGK_*` leaf entries. Changes propagate through this naming chain.

**Tournament system:** Multi-threaded, runs N rounds where each round generates a random card set (8 random units from the 105-unit pool + the 11 base set), then plays game pairs (color-swapped) between configured players. Results output to HTML files.

**AWS deployment:** `launch_tournament.sh` creates EC2 Windows instances that download exe + config from S3, patch the config (enable specific tournament, set rounds/threads), run the tournament, upload results to S3, and auto-terminate.

## 5. Relevant Existing Patterns & Conventions

### Config Entry Composition Pattern

The config system uses a naming/reference chain. Here's the full chain from player to the parameter under test:

```
PrismatAI_AB
  └─ MoveIterator: "HardIterator" (includes "BaseIterator")
       └─ ActionBuy slot: ["BuyComboGreedyAttack", "BuyComboGreedyWill", "BuyComboGreedyDefense"]
            └─ BuyComboGreedyAttack: ActionBuy_Combination of ["BuyGK_AttackValue", "BuyGK_WillScore", ...]
                 └─ BuyGK_AttackValue: ActionBuy_GreedyKnapsack with heuristic + filter + legacy flag
                      └─ _frontlinePenalty = (legacy ? 100000.0 : 5.0)  ← THE VARIABLE UNDER TEST
```

Creating a new player variant means duplicating the ENTIRE chain: new `BuyGK_*` leaf entries → new `BuyComboGreedy*` combinations → new `BCG*_Root` combinations → new iterators → new player entry. The plan correctly identifies all 13 new config entries needed.

### Legacy Pattern

`"legacy": true` on a `BuyGK_*` entry triggers THREE changes simultaneously:
1. Reverts heuristic function to `BuyAttackValue` (instead of `BuyAttackValue_Improved`)
2. Reverts heuristic function to `BuyBlockValue` (instead of `BuyBlockValue_Improved`)
3. Sets `_frontlinePenalty = 100000.0` (instead of `5.0`)

This bundling is the entire motivation for the plan — there's currently no way to change ONLY the frontline penalty value without also changing the heuristic functions.

### Tournament Skip Pattern

The existing `_skipColorSwap` boolean in Tournament demonstrates the pattern: parsed from config JSON, stored as a member, checked in `playRound()`. The plan reuses this exact pattern for `_skipNonFrontline`.

### AWS Launch Script Pattern

`launch_tournament.sh` creates a PowerShell user-data script that:
1. Downloads exe + config + weights from S3
2. **Hardcodes** the tournament name `"NeuralAB_vs_Original"` in sed-like line matching
3. Enables that tournament, disables all others, patches rounds/threads
4. Runs N processes, syncs results to S3, auto-terminates

The hardcoded tournament name is what Phase 3 of the plan needs to parameterize.

## 6. Current State & Known Issues

### What Works Today

- Tournament system runs reliably on local machine and AWS EC2
- `PrismatAI_AB` (neural eval, modern heuristics, penalty=5.0) achieves 51.9% WR vs `OriginalHardestAI` (2,016 games)
- `PrismatAI_AB_Legacy` exists but uses ALL legacy behaviors (old heuristics + penalty=100000 + legacy breach)
- AWS spot fleet launching and S3 result collection are proven workflows

### What Doesn't Exist

- No way to set `frontlinePenalty` independently via config — it's derived from the `legacy` boolean
- No way to filter tournament games by whether frontline units are in the random set
- `launch_tournament.sh` can only run the hardcoded `NeuralAB_vs_Original` tournament

### The Frontline Penalty Mechanism (What's Being Tested)

In `BuyKnapsackCompare::operator()` (the buy-phase card sorting comparator), when a card is **frontline** (can be targeted by enemy even behind blockers) AND enemy attack potential exceeds the card's health:

```cpp
if (c1.isFrontline() && _enemyAttackPotential >= c1.getStartingHealth())
{
    h1 /= _frontlinePenalty;   // Divide the card's heuristic value by the penalty
}
```

- **penalty = 5.0** (modern): Mildly discourages frontline purchases — the card needs to be 5x more valuable than alternatives to be bought
- **penalty = 100,000** (legacy): Effectively bans frontline purchases when enemy can kill them

The value `5.0` was introduced in commit `bc962f8` (Feb 13) without tournament validation. This experiment determines whether that change helped or hurt.

### Frontline Units in the Game

11 out of ~105 units have the "undefendable" (frontline) property in `cardLibrary.jso`. These include: Wild Drone, Galvani Drone, and various combat units. With 8 random cards drawn per game set (from ~94 non-base units), the probability of at least one frontline unit appearing is approximately:

P(at least 1 frontline) = 1 - C(83,8)/C(94,8) ≈ 59-62%

The plan estimates ~83% hit rate; the actual figure depends on the exact pool size and whether base-set units are included in the random draw.

### The `5.0` Value's Origin

The modern penalty value of `5.0` was set in a heuristic improvement commit (Feb 13) alongside other buy-function changes. It was chosen heuristically, not empirically validated. The legacy value of `100,000` effectively makes frontline units unpurchasable when enemy attack exceeds their health — a very conservative strategy that may miss opportunities where buying a frontline unit is correct.

## 8. Scope Boundaries

### Out of Scope

- **Changing the penalty value to something other than 5.0 or 100,000** — this is a binary comparison, not a sweep
- **Testing the Improved heuristic functions** — those are held constant (always Improved) in both test arms
- **Testing breach behavior** — BreachGreedyKnapsack is always modern (non-legacy) in both arms
- **Modifying OriginalHardestAI** — the baseline opponent is unchanged and uses its own legacy config chain

### Fixed Constraints

- **AWS spot c5.2xlarge** — the standard eval instance type, well-tested
- **NeuralNet evaluation** — both test players use the same 256h/3L model
- **7s think time** — matches the standard tournament configuration
- **8 random cards per set** — the standard Prismata game format

### Accepted Trade-offs

- Games without frontline units in the random set are skipped (wasted compute). The plan estimates ~17% waste; actual waste may be ~38-41% depending on exact pool size.
- The experiment tests penalty=5.0 vs penalty=100,000 (the two existing values) rather than searching for an optimal value.

---

## 9. Success Criteria

1. **~5,000 valid games per arm** (games where frontline units are present in the random set)
2. **Win rate comparison** with confidence intervals:
   - If WR difference < 3%: penalty value doesn't meaningfully affect performance → keep 5.0
   - If WR(5.0) > WR(100K) by >3%: 5.0 is genuinely better → validated
   - If WR(100K) > WR(5.0) by >3%: 100K is better → revert the change
3. **CI at 5,000 games: ±1.4%** — sufficient to detect a 3% effect
4. **Total cost under $5** — 12 spot instances × ~2hr × $0.14/hr ≈ $3.36

## 10. Key Questions for Reviewers

1. **Is the experimental design sound?** Both arms play against `OriginalHardestAI` (the baseline). Is this the right opponent for detecting a frontline-penalty effect, or would head-to-head (5.0 vs 100K playing each other) be more sensitive? What are the statistical considerations?

2. **Is `SkipNonFrontline` the right approach for filtering?** An alternative is to run all games and post-hoc filter results to only count games where frontline units appeared. The plan modifies the game loop itself. What are the tradeoffs?

3. **Is the sample size sufficient?** 5,000 games gives ±1.4% CI. If the true effect is 1-2%, we'd miss it. Is that acceptable, or should the experiment be larger?

4. **BuySafeguard chain contamination:** The plan notes that `BuySafeguard` references `BuyGK_AttackValue` (which has penalty=5.0). The FLLegacy config entries use `BuySafeguard` directly (not a FLLegacy variant). Does this contaminate the experiment — is the Safeguard sub-chain important enough that leaving it at penalty=5.0 undermines the isolation?

5. **Is the C++ change over-engineered or under-engineered?** The plan adds `FrontlinePenalty` as a general-purpose config key (usable by any future config entry), but the immediate need is one experiment. Is this the right level of investment, or should the approach be simpler/more general?

## 11. Glossary / Domain Terms

| Term | Definition |
|---|---|
| **Frontline** (undefendable) | A unit that can be targeted by enemy attack even when you have blockers. Most units are "behind the wall" — frontline units are exposed. Represented by `"undefendable": 1` in cardLibrary.jso, queried via `CardType::isFrontline()`. |
| **PartialPlayer** | A modular AI component that handles one phase of a turn (Defense, Ability, Buy, Breach). Multiple PartialPlayers compose into a full turn policy. |
| **ActionBuy_GreedyKnapsack** | The buy-phase PartialPlayer that uses a heuristic function to sort purchasable units by value-per-cost, then greedily buys the best affordable option. Contains the `_frontlinePenalty` field. |
| **BuyComboGreedy*** | An `ActionBuy_Combination` wrapper that runs multiple `GreedyKnapsack` instances in sequence (e.g., try AttackValue heuristic first, then WillScore as fallback). |
| **PPPortfolio / MoveIterator** | A portfolio of PartialPlayer combinations that generates candidate moves for the search algorithm. Each combination produces one full turn's actions. |
| **Legacy mode** | A flag (`"legacy": true`) that preserves the original academic AI behavior for baseline comparison. Bundles heuristic function reversion + frontlinePenalty=100,000. |
| **OriginalHardestAI** | The unmodified baseline AI (playout evaluation, legacy heuristics). The standard opponent for measuring model/heuristic improvements. |
| **PrismatAI_AB** | The project's best AI player: Stack Alpha-Beta search + NeuralNet evaluation + modern heuristics (including penalty=5.0). |
| **Random card set** | Each tournament game uses 8 randomly chosen units (from the ~94 non-base pool) plus the 11 base set units. This means not every game includes frontline units. |
| **Win rate (WR)** | Percentage of games won. Measured via round-robin tournaments (color-swapped pairs). CI computed from binomial proportion. |
| **frontlinePenalty** | A divisor applied to a frontline unit's heuristic buy value when enemy attack ≥ the unit's health. penalty=5.0 makes it 5x less attractive; penalty=100,000 effectively bans the purchase. |
| **Spot instance** | AWS EC2 instance using spare capacity at ~60-80% discount. Can be terminated by AWS with 2-minute warning. Tournament results are periodically synced to S3, so partial results are preserved. |
| **SkipColorSwap** | Existing tournament flag that skips the color-swapped game pair (used for self-play where both sides are identical). The plan models `SkipNonFrontline` on this pattern. |
| **BuySafeguard** | A fallback buy combination that acts as a "safety net" in the buy phase — avoids economic waste and ensures reasonable purchases when the primary heuristic doesn't find good buys. References `BuyGK_AttackValue` directly. |
