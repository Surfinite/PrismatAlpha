# Context Document: PrismataAI GPU Training Plan

*For reviewers with no prior knowledge of this project. Provides all context needed to critique the training plan at `docs/plans/2026-02-18-t4-training-plan.md`.*

---

## 1. What Is This Project?

**PrismataAI** is an AI system for **Prismata**, a two-player turn-based perfect-information strategy card game (no hidden information, no randomness after setup). Think chess-like decision complexity but with a branching factor of ~5 possible actions per move and games lasting ~70 player-turns.

The AI combines:
- **Alpha-Beta search** (or UCT/MCTS) for move selection
- **Neural network position evaluation** to score board states during search (replacing hand-crafted heuristics)

The neural net acts as the evaluation function inside the search tree — it takes a game state and outputs a value in [-1, +1] predicting who will win. It does NOT play moves directly (no end-to-end RL). This is the same role as the value head in AlphaZero, but we use it with a traditional search algorithm rather than MCTS+self-play for move selection.

## 2. The Neural Network

### Architecture
- **Input:** 1,785-dimensional feature vector (161 unit types × 11 features each + 14 global features like resources, turn number)
- **Trunk:** Linear projection → N residual blocks (Linear → LayerNorm → ReLU → Dropout → Linear → LayerNorm + skip connection)
- **Value head:** Linear(hidden_dim, hidden_dim/4) → ReLU → Linear(hidden_dim/4, 1) → optional tanh
- **Policy head:** Exists but unused (13% accuracy — too weak to help). Value-only training mode ignores it.

### Current Best Model: E2b
- hidden_dim=256, num_layers=2, dropout=0.1
- 738,945 parameters total
- Trained with: LR=1e-5, MSE loss, batch_size=512, tanh activation, label smoothing 0.95
- Adam optimizer with 5-epoch linear warmup then cosine decay

### Key Metric
Win rate vs **OriginalHardestAI** (the strongest hand-crafted AI, using Alpha-Beta search with a manually-tuned heuristic evaluation). This is the standard benchmark.

## 3. Training Data

### What the data is
Self-play games between two copies of OriginalHardestAI (the hand-crafted AI, NOT the neural net). Each game produces ~37 labeled positions (one per player-turn for both sides). Each position is labeled with the eventual game outcome (+1 win, -1 loss).

**Important:** The neural net is NOT used during data generation. It's purely supervised learning on game outcomes from the hand-crafted AI playing itself. This is AlphaZero "iteration 0" — bootstrap from an existing strong player.

### Data format
Binary shards, each containing a 64-byte header + N records of 7,152 bytes each (1,785 floats for state + metadata). Loaded via memory-mapped streaming for datasets too large for RAM.

### Data volume
| Milestone | Games | Records | When |
|-----------|-------|---------|------|
| Expert replays (human) | ~13K | ~251K | Pre-project (online games from rated players) |
| Self-play iteration 1 | 10K | ~370K | Feb 13, 2026 |
| Self-play iteration 2 | 63K | ~2.3M | Feb 15, 2026 |

Data is generated by a fleet of cloud VMs (AWS EC2 + GCP Compute Engine) running the C++ game engine. Production rate: ~4 games/min per 4-thread process across ~50 cloud instances.

## 4. Training History and Key Findings

### The Overfitting Problem
Every training run shows the same pattern: validation loss improves for a few epochs, then degrades while training loss continues dropping. The model memorizes rather than generalizes.

### V1 Experiments (Feb 13-15) — Failed
3 runs on 63K games. All showed best validation at epoch 1, with train accuracy hitting 98%+ by epoch 3-4. Val accuracy ceiling ~75%. No improvement over baseline.

**Root causes identified:**
1. **Training/inference mismatch (CRITICAL):** Training used raw MSE logits but C++ inference applied tanh(). The model never learned the right output scale.
2. **Learning rate 30x too high:** We used 3e-4; the reference paper (Churchill 2019) used 1e-5.
3. **Overparameterization:** 512h model (2.4M params) with only 63K games = terrible data/param ratio of 0.058:1.
4. **Temporal correlation:** ~37 positions per game are highly correlated, reducing effective sample diversity.

### V2 Experiments (Feb 17) — Success
9 experiments across 3 phases (loss function, LR sweep, data & capacity). Fixed the tanh mismatch, dropped LR to 1e-5, tested 256h vs 512h.

**Results:**
| Model | Params | Data:Param | Val Loss | Brier | Tournament WR |
|-------|--------|-----------|----------|-------|---------------|
| E2b (256h) | 739K | 0.085:1 | 0.338 | 0.089 | **26.7%** (1,008 games) |
| E1b (512h) | 1.08M | 0.058:1 | ~0.31 | — | 19.6% (1,008 games) |
| Pre-fix (512h) | 2.4M | 0.026:1 | — | — | 3.6% (1,120 games) |

V2 fixes gave a **5-7x WR improvement** (3.6% → 26.7%). Key finding: smaller model (256h) outperformed larger (512h) because the data/param ratio was better.

### Reference Point: Churchill (2019)
The original researcher achieved **58.8% WR** vs playout evaluation with:
- 500K self-play games
- ~1.1M parameter model
- Data/param ratio: ~0.45:1

Our current ratio at 356K games / 739K params = **0.48:1** — essentially at parity with Churchill's ratio. This is the strongest argument for expecting significant gains from retraining.

## 5. What Changed Since Last Training

The model was last trained (E2b) on **63K games** (2.3M records). We now have **356K games** (13.2M records). That's a **5.7x increase in training data** with no other changes.

This matters because:
1. The data/param ratio improved from 0.085:1 to 0.48:1 (into Churchill's range)
2. More unique game positions should reduce overfitting
3. The temporal correlation problem is diluted (more independent games)
4. The 512h model might now have enough data to outperform 256h

## 7. The Training Pipeline

### Software
- `training/train.py` — PyTorch training script with 30+ CLI flags
- Supports: streaming data loading (memory-mapped), CUDA/XPU/CPU, early stopping, step-level and epoch-level checkpointing, experiment JSON logging
- `training/export_weights.py` — converts PyTorch checkpoint to C++ binary format
- `aws/launch_training.sh` — launches AWS EC2 GPU instance, trains, uploads results to S3, auto-terminates

### Workflow
1. Launch GPU instance (GCP or AWS)
2. Instance downloads training code + self-play data from S3
3. Runs training with specified hyperparameters
4. Uploads model checkpoints + metrics to S3
5. Instance auto-terminates (no ongoing charges)
6. Download best model, export weights, deploy to C++ engine
7. Run tournament (1,000+ games) to measure WR

### Key Training Flags
| Flag | Default | Purpose |
|------|---------|---------|
| `--hidden-dim` | 512 | Model capacity (we use 256) |
| `--lr` | 3e-4 | Learning rate (we use 1e-5) |
| `--streaming` | off | Memory-mapped data loading for large datasets |
| `--value-only` | off | Skip policy head training |
| `--tanh-in-training` | off | Match C++ inference activation |
| `--patience` | 10 | Early stopping epochs |
| `--eval-every-steps` | 0 | Fine-grained checkpoint interval |
| `--device` | auto | Force cuda/xpu/cpu |

## 8. What the Plan Proposes

**15 training runs across 6 phases**, prioritized by expected impact:

1. **Baseline retrain** (1 run): Same winning E2b config on 5.7x more data. Highest expected value.
2. **Capacity sweep** (3 runs): Test 128h/384h/512h now that data ratio supports larger models.
3. **LR refinement** (3 runs): Sweep around 1e-5 since more data may shift the optimum.
4. **Regularization** (4 runs): Dropout, weight decay, label smoothing variations.
5. **Batch size** (2 runs): Larger batches may help on GPU with 24GB VRAM.
6. **Tournament eval**: Top 2-3 models play 1,000+ games to measure actual WR improvement.

Estimated total cost: < $2.00. Estimated total GPU time: ~40 minutes on L4.

## 9. Questions a Reviewer Might Want to Consider

1. **Is the phased approach right?** We sweep one hyperparameter at a time, carrying the winner forward. An alternative would be a random/grid search over all hyperparameters simultaneously. With only ~15 runs budgeted, is sequential more efficient than random?

2. **Are we varying the right things?** The plan holds architecture constant (2 residual blocks, ReLU, LayerNorm) and only varies width, LR, regularization, and batch size. Should we also test depth (3-4 layers), activation functions, or optimizer choice?

3. **Is the data/param ratio analysis sound?** We use Churchill's 0.45:1 as a reference. But Churchill's model architecture differs (details in his paper). Is the ratio comparison valid across different architectures?

4. **Is 356K games enough?** Churchill used 500K. Our data is still growing (~5K games/day from fleet). Should we wait for 500K before the big training push, or is 356K sufficient to learn from?

5. **Is the evaluation methodology robust?** Win rate over 1,000 games against a single opponent (OriginalHardestAI). Is this enough games for statistical confidence? Should we also evaluate against other opponents or use Elo estimation?

6. **Are there training techniques we're missing?** No data augmentation (game is asymmetric so position mirroring is invalid). No curriculum learning. No learning rate scheduling beyond warmup+cosine. Would any of these help?

7. **Is value-only the right choice?** The policy head is at 13% accuracy and disabled. Self-play data contains game outcomes (for value) but not move labels (for policy). Would adding policy learning via move prediction help the value head indirectly?

8. **Overfitting mitigation:** Previous runs overfit within 10-20 epochs regardless of config. With 5.7x more data, will the pattern simply shift later, or do we need architectural changes (e.g., more aggressive dropout, batch normalization, mixup augmentation)?

## 12. Glossary

| Term | Meaning |
|------|---------|
| **WR** | Win rate (% of games won in tournament) |
| **OriginalHardestAI** | The strongest hand-crafted AI. Uses Alpha-Beta search with Will Score heuristic evaluation. The standard benchmark opponent. |
| **E2b** | Experiment 2b from V2 hyperparameter sweep. Current best model (256h, 26.7% WR). |
| **hidden_dim / Nh** | Width of hidden layers in the neural network (e.g., 256h = 256-dimensional hidden layers) |
| **Data:Param ratio** | Number of training games divided by number of model parameters. Higher = less overfitting risk. |
| **Brier score** | Calibration metric: mean squared error between predicted win probability and actual outcome. Lower is better. |
| **Val loss** | Validation set loss (MSE between predicted value and game outcome). Lower is better. |
| **Tanh mismatch** | A bug where training used raw logits but C++ inference applied tanh(), causing a systematic prediction error. Fixed in V2. |
| **Streaming** | Memory-mapped data loading that avoids loading the full dataset into RAM. Required when dataset > 32GB RAM. |
| **Self-play** | Games where two copies of the same AI play each other. Generates training data labeled by game outcome. |
| **Preemptible/Spot** | Discounted cloud instances that can be reclaimed by the provider. 60-80% cheaper but may be interrupted. |
| **L4** | NVIDIA L4 GPU (Ada Lovelace, 24GB VRAM). ~3.7x faster than T4 for FP32 training. |
| **T4** | NVIDIA T4 GPU (Turing, 16GB VRAM). Available on AWS g4dn instances. |
