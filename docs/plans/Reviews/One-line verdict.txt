**One-line verdict:**  
This is a thoughtful, research-backed plan that correctly identifies a two-stage LLM pipeline as the quality/cost sweet-spot for post-game commentary, but it over-engineers around hallucination checks and batch infra while under-specifying the actual narrative prompt and style-control mechanics that will determine whether the prose feels human.

---

**What’s good (keep as-is)**  
- Two-stage split (analysis → narrative) is the right call; the WSC Sports and arXiv evidence is solid and matches the manual workflow that already produced great output.  
- Re-using the proven Batch API / polling / resume pattern from `discord_knowledge_extractor.py` is low-risk and gives the 50 % discount for free.  
- Using resource-validated buys as ground truth and only the random-set unit knowledge in-context keeps token count and hallucination surface low.  
- Haiku default + Sonnet opt-in is the correct cost/quality knob for a solo dev who already had a $805 AWS shock.  
- Intermediate JSON caching (data + analysis) is cheap insurance against re-running the C++ step.

---

**Concerns & risks (ranked)**  
1. **Narrative prompt under-specified** – the plan only says “feed analysis JSON + selected notable turns + few-shot examples” but never defines how the LLM is instructed to *sound*. The difference between “analytical”, “hype”, and “casual” is not a flag, it’s prompt engineering that isn’t written yet.  
2. **Few-shot token bomb** – 3–4 k tokens of pasted commentary per call can push a 15-turn game to ~8 k input before the actual analysis is even inserted. This erodes the Haiku cost advantage and risks hitting the 8 k limit the plan warns about.  
3. **Hallucination checks are theatre** – verifying turn numbers and unit names catches obvious lies but not the subtle ones that break credibility (“T12’s double Conduit was *the* pivotal moment” when eval delta was only 3 %, etc.). A cheap LLM judge call (“Does every claim appear in the analysis JSON?”) would catch more with negligible cost.  
4. **No escape hatch for stepper-unreliable games** – if the C++ stepper applied <80 % of clicks the buys are guesses, yet the pipeline still presents them as ground truth. Should fall back to eval-only mode automatically.  
5. **Batch resume is fragile** – the plan copies the extractor’s JSON checkpoint pattern but the extractor only has one stage; here Stage 2 depends on Stage 1 output. A half-finished batch will leave orphaned analysis JSONs that the resume logic must validate before continuing.

---

**Suggested changes (actionable)**  
A. **Narrative prompt v0** – add a real template (keep under 1 k tokens) that explicitly instructs the model to:  
   - open with a 1-sentence hook,  
   - use present-tense play-by-play for turning-point turns,  
   - end each Discord message with a forward-reference to keep readers clicking.  
   Store it in `tools/prompts/narrative_user_template.md` and version it like code.  

B. **Dynamic few-shot** – instead of always dumping 1–2 full commentaries, keep a 500-token “style distillation” paragraph extracted from each example (keep the voice, drop the game-specific prose). Include the full example only when the game shares a key unit or upset pattern.  

C. **LLM judge verification** – after Stage 2, fire a 200-token Haiku call:  
   “List every factual claim in the commentary that is not directly supported by the analysis JSON.”  
   If the list is non-empty, regenerate once; still fails → prepend “[auto-gen]” to the file name so humans know.  

D. **Auto fallback for stepper-unreliable** – if `stepper_applied_pct < 0.8` switch to `--eval-only` mode and add a footnote: “Clicks could not be fully validated; narrative uses neural eval only.”  

E. **Batch dependency check** – on `--resume-batch`, verify that every analysis JSON expected by the narrative batch actually exists and is valid; if not, delete the orphaned batch status and restart from Stage 1.

---

**Alternatives worth considering**  
- **Single-stage with forced outline** – ask Haiku to emit a hidden JSON outline first, then continue with prose in the same completion. Costs one call, keeps context hot, and the outline can be programmatically verified.  
- **Retrieval over few-shot** – instead of in-context examples, embed the 3 commentaries and retrieve the most stylistically similar paragraph(s) per game. Adds a ~300-token retrieval step but keeps prompt small.  
- **Local small model for Stage 1** – a 3B model fine-tuned on 30k Prismata positions can spit out the JSON analysis for pennies; keep Claude for the prose stage only. Requires GPU time but breaks the API cost curve entirely.

---

**Additions (missing from plan)**  
- **Unit synergy appendix** – a 200-token mini-section in the narrative prompt that lists known two-card synergies for the random set (e.g., “Plasmafier + Galvani Drone”) so the model can spot combos instead of just single-unit value.  
- **Time-pressure lens** – the replay JSON contains `timeUsed` per turn; surface “who was in time-trouble on the critical turn” as an explicit field in the analysis JSON—human commentators always mention clock tension.  
- **Discord emoji shorthand** – allow the prompt to use `:unit_name:` placeholders that get replaced with Discord custom emojis already in the server (keeps messages shorter and visually consistent).  
- **Progressive disclosure CLI flag** – `--depth casual|expert|master` that controls how much eval jargon is kept in the final prose (e.g., drop “eval dropped 12 %” at casual, keep it at master).

---

**Removals (drop from plan)**  
- **Phase 6 A/B compare** – premature until the narrative prompt is stable; one good style is enough for launch.  
- **Explicit “hype” vs “casual” style flags** – remove until real prompts exist; otherwise it’s dead code.  
- **Token-count hard limit warnings** – the 8 k cap is arbitrary; Haiku context is 200 k. Keep an eye on cost, not token count.

---

**Minor / nits**  
- Rename `generate_postgame_commentary.py` to `postgame_commentary.py` to match the terse naming convention of sibling tools.  
- Use `xz` compression for cached JSON – 30 % smaller than gz for the repetitive replay data.  
- Add `--dry-run` that prints the exact prompts and token counts without calling the API – speeds up prompt iteration.

---

**Assumptions I’m making (validate in meta-review)**  
- The C++ exe already outputs `stepper_applied_pct` in its JSON; the plan mentions it but I can’t see the schema.  
- The 3 existing commentaries in `bin/commentary/` are representative of the voice the community wants; no hidden duds.  
- Anthropic Batch API still allows prompt-caching prefixes inside each individual request (plan assumes yes).
