# Post-Game Replay Commentary Pipeline Review

## 1. One-line verdict
**Solid, well-researched plan with strong research backing and pragmatic cost focus, but overengineered two-stage pipeline and premature batching risk solo dev distraction from core single-game MVP.**

## 2. What's good
- **Research-driven decisions** — WSC Sports, arXiv survey, Anthropic docs citations make the two-stage + structured data + few-shot choices credible, not arbitrary.
- **Extends existing tools surgically** — Modifying `generate_commentary_data.py` with `--json-output` flag preserves existing functionality perfectly (Phase 1 anti-pattern guards are excellent).
- **Cost obsession is spot-on** — $0.01-0.02/game target, Haiku default, batch+caching math, sync vs batch distinction all show battle-tested API cost awareness.
- **Programmatic verification pass (Phase 2c)** — Checking turn numbers/unit names/eval against ground truth data is the right hallucination defense; cheap and effective.
- **Caching strategy (Phase 4d)** — `data_`, `analysis_`, `commentary_` files enable prompt iteration without C++ re-runs. Critical for solo dev tuning.
- **Proven patterns** — Following `discord_knowledge_extractor.py` batch pattern eliminates reinventing wheels.
- **Clear verification checklists** — Every phase has concrete test commands with the 3 known-good replays. Makes testing foolproof.
- **Schema-first design** — `commentary_schema.json` + structured analysis output creates parseable intermediates for debugging.
- **Realistic success criteria** — Factual accuracy >95%, quality parity with manual examples, works without C++ are all achievable and measurable.

## 3. Concerns & risks (ranked by severity)

**HIGH severity:**
1. **Two-stage LLM unnecessary complexity** — For 15-turn games producing ~4 Discord messages, single Haiku call with structured data + few-shot examples will likely hit 90% of quality at half cost/latency. Research citations are sports video (WSC) which does full highlight reel extraction; we need 500-word Discord post. Test this first.
2. **Batch Phase 5 before single-game MVP** — 100-game batches when you have ~30K replays available? Solo dev time sink on checkpointing/resume/polling when core deliverable is "one command → one commentary file".
3. **Unit knowledge extraction brittle** — Text search in 1,266-line `03-advanced-units.md` for headers will break on formatting changes, partial matches, or missing entries. No fallback plan specified beyond "mergedDeck stats".

**MEDIUM severity:**
4. **C++ `--analyze` unreliability unaddressed** — Plan acknowledges 300s timeouts + x86 OOM but doesn't specify fallback beyond `--eval-only` flag. What % of games fail analysis entirely?
5. **No ablation on few-shot value** — Claims "dramatically improve narrative quality" but no evidence these 3 specific examples generalize beyond their games. 3-4K token cost per call is 30% budget hit.
6. **Quality rubric incomplete** — Phase 6 only checks factuals/structure. No coverage/insight metrics (does it mention biggest eval swing? Decisive factor from analysis?).

**LOW severity:**
7. **Prompt token estimates optimistic** — Analysis "~4K in/2K out" ignores full condensed rules (1,200) + unit knowledge + game data. Realistically 6-8K in.
8. **No error recovery for LLM failures** — What if structured analysis JSON is malformed despite schema enforcement?

## 4. Suggested changes

**Phase 1 (Data extraction):**
- Add `random_set_unit_knowledge` fallback: if KB lookup fails, populate with `mergedDeck` stats + 1-line auto-generated description using base unit similarity (Drone-like? Engineer-like?).
- Add `game_characteristics` top-level JSON field: `{"length_category": "short|medium|long", "rating_diff": 366, "upset": true, "has_time_pressure": true}`. Used for dynamic few-shot selection.

**Phase 2 (Analysis):**
- **Convert to optional flag**: `--analysis-stage` instead of mandatory. Default to single-stage narrative that does light analysis inline. Prove two-stage value first.
- Add `fail_fast` verification: if >10% turns have missing eval_delta or stepper_reliable=false, skip analysis stage entirely.

**Phase 4 (CLI):**
```
# NEW PRIORITY ORDER - do these first, in sequence:
1. python tools/generate_postgame_commentary.py "FxCfR-K49T+" --single-stage  # Test hypothesis
2. python tools/generate_postgame_commentary.py "FxCfR-K49T+"               # Two-stage if single fails quality
3. --batch support only AFTER single-game perfected
```

**Phase 5 (Batch):**
- Defer to post-MVP. Replace with simple parallel sync: `parallel -j4 python tools/generate_postgame_commentary.py ::: $(cat codes.txt)`. 4x throughput, no batch complexity.

**General:**
- Add `--dry-run` flag: generate prompts without API calls, save to `prompts_debug/{CODE}_analysis_prompt.txt`. Essential for token counting/verification.
- `--max-messages 6` flag capping output messages (prevent LLM bloat).

## 5. Alternatives

| Current Plan | Alternative | When to use |
|--------------|-------------|-------------|
| **Two-stage LLM** | **Single-stage with inline analysis**<br>`"First, analyze in JSON format: {schema}. Then write commentary using that analysis:"` | Default. Half cost, 90% quality. Test first. |
| **Text search KB** | **Pre-process KB into JSON index**<br>`unit_knowledge.json: {"Plasmafier": {...}}` | Production. No runtime parsing. |
| **Batch API** | **Parallel sync**<br>`parallel -j4` or `asyncio.gather()` | <50 games. Simpler, instant results. |
| **3 manual few-shots** | **Style-distilled system prompt**<br>Extract patterns from examples into 500-token system prompt | All games. Saves 3-4K tokens/call. |
| **Haiku structured → freeform** | **Sonnet single-pass** | Budget unlimited, max quality needed. |

**Strong recommendation: Start with single-stage Haiku + test on all 3 replays vs manual quality. Only add analysis stage if <90% quality.**

**Phase 6 quality rubric metrics:**
```
FACTUAL: 10pts (turn existence 4, units in deck 3, winner correct 2, eval match 1)
COVERAGE: 5pts (mentions biggest swing 2, decisive factor 2, 3+ specific turns 1)
STRUCTURE: 3pts (delimiters 1, <2K chars 1, 4-8 messages 1)
SPECIFICITY: 2pts (# turn refs + # unit mentions)
```

**New CLI flag:** `--quality-threshold 90` — Fail if rubric < threshold.

## 7. Removals

- **Phase 5 entirely** — Batch processing. Use `parallel` or `asyncio` instead. Premature optimization.
- **Dynamic few-shot selection logic** (Phase 3a) — Use shortest example always (`WjhmP-WWdXx.txt`). Simpler, sufficient.
- **`--style hype/casual`** — No evidence community wants this. Stick to `analytical` matching manual examples.
- **Sonnet in batch estimates** — Irrelevant if batch removed.

## 8. Minor / nits

- Model names inconsistent: "Haiku 4.5" vs actual `claude-haiku-4-5-20251001`. Use SDK constants.
- Token estimates should specify in/out separately per stage with cache hit assumptions.
- Phase 2c verification should log *which* claims failed, not just pass/fail.
- No mention of replay code sanitization (`+`→`_PLUS_`, `@`→`_AT_`) in filenames — assume `generate_commentary_data.py` handles.
- `--think-time 50` CLI flag documented but no implementation details.
- Cost table says "Batch (Sonnet 4.5)" but Sonnet batch discount math missing.

## 9. Assumptions you're making

1. **`generate_commentary_data.py` stdout is parseable JSON** in `--validate`/`--eval` modes for integration testing.
2. **C++ `--analyze` JSON schema stable** — assuming fields like `eval_pct`, `ai_buys` won't change.
3. **`bin/commentary/*.txt` files readable as-is** for few-shot extraction (no weird encoding).
4. **ResourceTracker accuracy sufficient** for `--validate-only` mode quality (plan acknowledges edge cases).
5. **3 test replays representative** — `FxCfR-K49T+` (upset), `WjhmP-WWdXx` (Masters), `uP8mG-tr75d` (grind) cover main game types.
6. **Anthropic SDK batch polling reliable** — assuming `discord_knowledge_extractor.py` pattern works unchanged.
7. **No rate limits hit** in sync mode — 2 calls/game × few games ok, but batch needed for scale.

**Validation needed:** Run single-stage POC on 3 replays vs manual quality before committing to two-stage complexity.
