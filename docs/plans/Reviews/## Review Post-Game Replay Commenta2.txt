## Review: Post-Game Replay Commentary Pipeline

---

### 1. One-line verdict

A well-researched, appropriately scoped plan that solves a real problem with sensible architecture — but it has meaningful gaps in error handling, prompt engineering detail, and the Phase 6 quality loop is too weak to actually drive iteration.

---

### 2. What's good

**Two-stage pipeline is the right call.** The justification is solid and grounded in real production systems. Don't second-guess this in response to reviewer question #1. A single-pass call *will* produce worse results — analysis requires system-level thinking, narrative requires rhetorical thinking. They fight each other in a single prompt.

**Phase 1 JSON schema is excellent.** Having `unit_knowledge` pre-extracted per-unit, `stepper_reliable` as a validity flag, and `ai_agrees` at the turn level gives the LLM exactly what it needs. The anti-pattern guard against importing anthropic SDK in Phase 1 is correct.

**Intermediate file caching (Phase 4d) is the right architectural decision.** This will pay dividends immediately — prompt iteration on the narrative stage without re-running C++ is the kind of practical affordance that makes solo development tolerable.

**Dynamic few-shot selection logic (Phase 3a) is well-thought-out.** Matching examples to game characteristics (upset → use Giant Killer, long game → use xYotsu) is a genuine quality improvement over static examples. This should be preserved and potentially expanded.

**The anti-pattern guards throughout are good.** Particularly: don't use structured output for narrative, don't dump raw turn data into narrative prompt, don't import with module-level side effects. These reflect real experience.

**Phase 5 batch design follows the proven pattern.** The `discord_knowledge_extractor.py` reference is appropriate, and the two-batch approach (analysis first, narrative second) correctly handles the dependency chain.

---

### 3. Concerns & risks

**[Severity: High] The verification pass (Phase 2c) is too shallow and will miss the most common hallucinations.**

The plan checks: turn numbers exist, unit names in deck, eval values ±2%. This will *not* catch:
- Invented player actions ("Surfinite bought a Steelsplitter on T7" when they bought a Drone)
- Fabricated time pressure claims ("Kolento was down to 12 seconds" when time data shows 45s)
- Wrong eval directionality ("P1 took a commanding lead on T6" when eval swung *against* P1)
- Invented AI agreement claims ("the AI concurred with this buy" when `ai_agrees` is false for that turn)

The plan notes in context document Section 6 that resource validation is "not 100% accurate" for `buySac` and complex ability interactions. The LLM will be given imperfect ground truth and the verification pass won't catch errors that originate in bad input data.

**Recommended fix:** Extend Phase 2c to also verify: every specific purchase claim cross-references the `buys` array for that turn, every time pressure statement is grounded in actual `time_bank` values, and every eval claim (directionality, magnitude) matches `eval_delta` sign and approximate magnitude. This is still programmatic — no extra LLM call needed.

---

**[Severity: High] The prompts are the most important part of this system and the plan barely specifies them.**

The plan creates three prompt files (`analysis_system.md`, `narrative_system.md`, `narrative_user_template.md`) but tells us almost nothing about their content beyond token budget targets and what knowledge to exclude. The entire quality of the output depends on these files, but they're described as "create new file" with no content guidance.

Contrast this with Phase 1, which has a complete schema with field-level specificity. Phase 2 and Phase 3 deserve the same treatment.

**Recommended fix:** The plan should include, at minimum:
- The key instructions for the analysis prompt (what constitutes a "turning point" vs. a routine turn, how to identify phases, what makes a decision "key")
- The narrative prompt's framing sentence and core stylistic directives
- The user message template structure with labeled placeholders

These don't need to be the final versions — they're direction for implementation, not a commitment.

---

**[Severity: High] The `--validate-only` mode (no C++ exe) produces fundamentally worse commentary and this isn't acknowledged.**

Section 9 lists "Works without C++ exe" as a must-have success criterion, but without `eval_pct`, `eval_delta`, `ai_buys`, and `ai_agrees`, the analysis stage is missing its most important inputs:
- No neural eval = can't identify turning points objectively
- No AI comparison = can't identify mistakes
- No AI-recommended buys = can't do "the AI preferred X, but player chose Y" analysis

Commentary from `--validate-only` won't be "still usable" — it'll be structurally different and significantly worse. The plan should explicitly document this degradation and consider whether `--validate-only` should produce a shorter, different format (e.g., 2 messages instead of 4, no mistake analysis).

---

**[Severity: Medium] The 8K token cap on the narrative stage input (Phase 3, anti-pattern guard) will be hit regularly and the recovery path is unspecified.**

A single few-shot commentary example is ~3,500 tokens. The analysis JSON will be ~2,000 tokens. Two examples + analysis + per-turn notable data easily exceeds 8K. The plan says "trim the less relevant few-shot example" but doesn't specify *how* that decision is made or what "trim" means — drop the whole example, or truncate it?

For long games (>25 turns like `uP8mG-tr75d` at 39 turns), the per-turn data alone could push past this limit.

**Recommended fix:** Specify the priority order for trimming: first drop the second few-shot example, then truncate per-turn data to only turns with |eval_delta| > threshold, then truncate the first few-shot example to first/last message only. Make this explicit and implement it as a function with a clear token-counting loop.

---

**[Severity: Medium] Phase 6 quality assurance is not a feedback loop — it's a one-time check.**

The "quality rubric" measures: factual accuracy, coverage of biggest eval swing, structure, and specificity. These are necessary but not sufficient. More critically, the plan describes no process for *acting on failures*. If the rubric shows 88% factual accuracy (below the 95% threshold), what happens? The plan says "human review" for prose quality and leaves factual failures implicit.

The `--compare` tool (Phase 6a) is useful but generates a diff — it doesn't tell you which variant is better or why.

**Recommended fix:** Phase 6 should include an explicit iteration workflow: rubric failure → identify failing claim type → locate which prompt section produced it → targeted prompt edit → re-run on all 3 test replays. Even a bullet list of this process would make Phase 6 actionable rather than decorative.

---

**[Severity: Medium] The `--style casual` flag (Phase 3d) directly contradicts the "non-beginner audience" out-of-scope declaration.**

Context document Section 8 explicitly states "Commentary for non-expert audiences — out of scope." A `--style casual` option for "accessible for non-expert audience" is scope creep inside the plan itself. This either needs to be removed or the out-of-scope declaration needs to be revised.

---

**[Severity: Low] The `subprocess vs import` decision for Phase 4b (calling `generate_commentary_data.py`) is unresolved.**

The plan says "subprocess or import" but the anti-pattern guard says "don't import if it has module-level side effects." This implies subprocess is the right answer, but subprocess has costs: no shared memory, serialization overhead, harder error propagation. For a 690-line script that the developer controls, refactoring to expose a callable function and eliminating module-level side effects is probably better than permanent subprocess coupling.

*Assumption: I'm guessing `generate_commentary_data.py` has module-level side effects (probably argument parsing or file I/O at import). If it doesn't, import is cleaner.*

---

**[Severity: Low] Prompt caching TTL mismatch in batch mode (Phase 5c).**

The plan notes "5-min cache (batch processes fast)" but Anthropic's batch API processes requests asynchronously over minutes to hours. The 5-minute ephemeral cache TTL means cache hits are **not guaranteed** in batch mode — if processing spills past 5 minutes (very likely for 15+ games), the cache will have expired and subsequent requests will pay full input cost. The 1-hour cache TTL (requires explicitly specifying it) is more appropriate for batch.

*Assumption: I'm relying on the context document's reference to Anthropic prompt caching docs and the noted 5-min/1-hr TTL distinction. The plan should verify which TTL is available for batch requests.*

---

### 4. Suggested changes

**Phase 2c: Expand verification checklist**
Add these programmatic checks:
- For every turn number T referenced in any analysis field: verify `turns[T].buys` and `turns[T].player_name` against the structured data
- For every "mistake" claim: verify `turns[T].ai_agrees == false` OR `eval_delta` is negative for that player
- For eval directionality claims: verify `eval_delta` sign matches the claimed direction

**Phase 3, narrative prompt design: Add a "grounding constraint" instruction**
The narrative system prompt should include an explicit instruction like: *"Only reference purchases confirmed in the `buys` arrays. Do not invent or infer purchases not present in the data. If a turn has empty `buys`, say the player passed or pressed end turn."* This is the single highest-value prompt engineering instruction for hallucination prevention.

**Phase 3a: Codify the few-shot selection as a decision function, not prose**
The current description is "if long game, if upset..." — this should be a documented function signature in the plan:

```python
def select_few_shot_examples(game_data: dict) -> list[str]:
    """Returns 1-2 commentary file paths based on game characteristics."""
```

with the selection logic explicit. This makes implementation unambiguous.

**Phase 4d: Add a `--force-regen` flag** to bypass caching for all stages. The plan mentions caching but doesn't mention override. When iterating on prompts, you'll always want to regenerate from cached data — but sometimes you'll need to regenerate the data too.

**Phase 5b: Make the two-batch dependency explicit in the CLI**
The batch pipeline should surface the intermediate state: after analysis batch completes, print "Analysis complete for N/M games. Submitting narrative batch..." before proceeding. This is not just UX — it's a natural checkpoint to inspect analysis quality before spending on narrative generation.

---

### 5. Alternatives

**Alternative to two LLM stages: One analysis call, one self-checking narrative call**

Instead of the current Analysis → Narrative split, consider: Analysis + Narrative as a single combined prompt (eliminating Stage 1), followed by a second call that is *solely* a verification/correction pass using the structured game data. This reorders the stages from "analyze then narrate" to "narrate then verify."

Advantage: The narrative can flow more naturally without the intermediate JSON representation losing nuance (e.g., a "turning point" that's emotionally significant but not the biggest eval swing). Disadvantage: More hallucination risk in the combined call. I'm not recommending this over the current approach — just noting it exists and the current plan's choice is defensible.

**Alternative to file-based few-shot examples: Extracted style guide**

The plan's Reviewer Question #3 asks about this. Rather than including full ~3,500-token commentary examples, extract a 400-500 token "style guide" capturing the patterns: sentence structures used, types of strategic observations made, how turn references are formatted, how player ratings are incorporated. This costs less tokens and is more transferable across game types.

The risk: you lose concrete examples of good commentary and rely on the model to internalize abstract style rules. For Haiku 4.5, concrete examples likely outperform abstract rules — keep the current approach for now, but consider the style guide extraction as a later optimization if token costs become a concern.

**Alternative to `--validate-only` mode: Remove it from must-have criteria**

As noted in Concern #3, `--validate-only` without neural eval produces structurally inferior commentary. Rather than shipping a degraded mode as a must-have, consider making it a future enhancement or limiting it to a short factual summary rather than full commentary format. The C++ exe requirement is not a significant barrier — it's already a dependency of the project.

---

### 6. Additions

**Missing: Handling for games with unreliable stepper data**

The context document notes that `stepper_reliable` (>80% click application rate) flags unreliable games. The plan references this field in the schema but has no specified behavior when `stepper_reliable = false`. The pipeline should either: warn and continue with degraded confidence, or skip the analysis stage's AI-comparison claims entirely and rely only on eval deltas. This should be an explicit code path, not a silent inconsistency.

**Missing: Handling for very short games (rushes, <8 turns)**

A rush game ending in 6-8 turns has very different commentary needs than a 40-turn grind. The current plan targets 4-8 Discord messages, which is wildly wrong for a 6-turn game. The plan should specify: if `total_rounds < 10`, use a short-game template (2 messages max) with different phase structure. The analysis prompt should know to treat it as a "speed game" rather than applying the Opening/Development/Midgame/Endgame phase model.

**Missing: Output validation before file write**

Phase 3c checks for delimiters and message length, but doesn't validate that the commentary is non-empty, non-repetitive, or actually complete (sometimes LLMs truncate mid-sentence if max_tokens is hit). Add: check that the last message ends with a complete sentence (simple heuristic: ends with `.`, `!`, or `?`), and that at least N messages were generated (where N ≥ 2 for a normal game).

**Missing: `max_tokens` specification for both stages**

The plan specifies token *estimates* for cost purposes but never specifies the `max_tokens` parameter for the API calls. For Stage 1 (structured output, ~2K out estimated): `max_tokens=2500` as a conservative ceiling. For Stage 2 (narrative, ~3K out estimated): `max_tokens=4096`. Getting this wrong causes silent truncation. The plan should pin these values.

**Missing: What happens when C++ subprocess fails or times out?**

The context document notes `--analyze` can timeout on 40+ turn games (300s limit). The plan for Phase 1 doesn't address this failure mode. Should the pipeline automatically fall back to `--eval-only` on timeout? This is a known issue that will definitely be hit on long games (like the 39-turn `uP8mG-tr75d` test replay). The fallback should be automatic with a logged warning, not a hard failure.

**Missing: Replay code validation before doing any work**

The pipeline should validate the replay code format (alphanumeric + `+` and `@`) before fetching from S3. A typo in the replay code will currently propagate through Phase 1 before failing. A simple regex check at the start saves API costs and produces a cleaner error.

---

### 7. Removals

**Remove `--style hype` and `--style casual` from Phase 3d scope.**

`--style hype` is low-value for a small community of expert players who will find it jarring. `--style casual` contradicts the out-of-scope declaration. These add prompt engineering complexity and test surface area without clear payoff. Ship `--style analytical` as the only style in the initial version. Add others after the core pipeline is validated. The plan correctly notes no timeline pressure — don't pre-build features that haven't been requested.

**Remove Phase 6a (`--compare` tool) from this plan.**

The A/B comparison tool is genuinely useful for prompt iteration but it's not part of building the commentary pipeline — it's a developer tool for tuning it afterward. It belongs in a separate "prompt engineering workflow" phase or as a future enhancement. Including it in Phase 6 makes Phase 6 feel like two things: quality assurance and developer tooling. Focus Phase 6 entirely on the quality rubric and test suite.

**Remove the `--batch-codes` inline CLI option (Phase 5a).**

Having two batch input methods (`--batch codes.txt` and `--batch-codes "A,B,C"`) is minor convenience for a solo developer who already has a text file workflow. The inline option adds argument parsing complexity and a CSV parsing edge case (replay codes contain `+` and `@` which need escaping). Support file input only.

### 8. Minor / nits

- Phase 4c cost tracking shows Sonnet priced at `~$0.08/game` but the global cost table shows `~$0.10/game`. Inconsistency — pick one and document the token assumptions clearly.

- The `commentary_schema.json` (Phase 1) has `"tier": "rare"` as an example value for random set units. The context document doesn't define a tier system with values like "rare" — it references tier *rankings* in the KB files. Verify this field actually exists in the data or remove it from the schema.

- Phase 4b says the pipeline orchestration step 1 calls `generate_commentary_data.py` "internally (subprocess or import)" — resolve this ambiguity in the plan, don't leave it for implementation time. Given the anti-pattern guard in the same section, it should say "subprocess" explicitly.

- The research source for arXiv 2506.17294 is cited in the plan but today's date is 2026-02-22. An arXiv paper numbered 2506.xxxxx would be from June 2026, which is in the future. Either the citation is a placeholder/hallucination or the numbering convention is being used differently. Worth verifying this source exists.

- The plan mentions `Sonnet 4.5` in the glossary as `claude-sonnet-4-5-20241022` but Haiku 4.5 is `claude-haiku-4-5-20251001`. The date suffix on Sonnet 4.5 (October 2024) predates Haiku 4.5 (October 2025), which seems inconsistent — Haiku 4.5 shouldn't be newer than Sonnet 4.5. Verify the model identifiers are correct before hardcoding them.

- Phase 2b defines `analysis_schema` with Python type annotations (`str`, `[int, int]`) mixed with JSON Schema style. Choose one representation and be consistent — if this is going into structured output mode, it needs to be a valid JSON Schema, not Python pseudocode.

### 9. Assumptions you're making

1. **`generate_commentary_data.py` has module-level side effects** (probably argparse setup or file path resolution at import time) — this is why the plan hedges on subprocess vs import. *If it doesn't, direct import is cleaner and the plan should say so.*

2. **The three test replays (`FxCfR-K49T+`, `WjhmP-WWdXx`, `uP8mG-tr75d`) are all fetchable and produce valid output from the current data extraction pipeline.** The plan treats them as ground truth for testing. If any of them has known issues (e.g., `uP8mG-tr75d` at 39 turns might hit the `--analyze` timeout), the test suite will fail unexpectedly.

3. **The `mergedDeck` JSON from Lunarch S3 contains enough information to populate all schema fields** (particularly `tier`, `abilities` as readable text, and `supply`). Some card properties might be encoded differently than the schema assumes.

4. **Claude Haiku 4.5 supports structured outputs (JSON schema mode).** The context document states "GA on Haiku 4.5" — I'm taking this at face value. If Haiku 4.5 structured output has reliability issues in practice (partial JSON, schema violations), the Phase 2c verification pass becomes load-bearing rather than belt-and-suspenders.

5. **The 5,090-line knowledge base in `docs/commentary-knowledge/03-advanced-units.md` has consistent unit name headers** that can be found via simple text search. If unit names in the KB use different formatting than in `mergedDeck` (e.g., "Tesla Coil" vs "TeslaCoil" vs "Tesla coil"), the Phase 1b unit knowledge lookup will silently fail to find entries.

6. **The existing 7 commentary files in `bin/commentary/` include clean final versions** (not intermediate drafts) suitable as few-shot examples. If some files are iteration artifacts with known quality issues, they shouldn't be used as examples without curation — the plan treats them as uniformly usable.
