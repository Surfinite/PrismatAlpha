## Review: Post-Game Replay Commentary Pipeline

### 1. One-line verdict

A well-researched, sensibly scoped plan that over-engineers certain areas (batch mode, A/B tooling, style variants) relative to the actual audience size and developer bandwidth, while under-specifying the hardest part — the prompts themselves.

---

### 2. What's good

- **Two-stage pipeline is well-justified.** The WSC Sports and academic citations are real and relevant, and the reasoning for why single-pass degrades is correct. The split between structured analysis and free-form narrative is the right call.

- **Phase 1 (structured JSON extraction) is the highest-value work.** Decoupling data extraction from LLM calls is excellent engineering. The JSON schema is well-thought-out and includes the right fields. This phase alone makes the whole project more maintainable and debuggable.

- **Intermediate file caching (Phase 4d).** Saving `data_*.json` and `analysis_*.json` separately is critical for a solo developer iterating on prompts. This is exactly the right design and should be implemented early, not as a Phase 4 afterthought.

- **Anti-pattern guards are specific and useful.** "Do NOT use structured output for narrative," "Do NOT import the anthropic SDK in Phase 1," etc. These show real experience with the failure modes and will prevent regressions.

- **Cost tracking and projection.** For someone who got a $805 AWS surprise bill, building cost awareness into every command is smart.

- **`--validate-only` fallback.** Ensuring the pipeline works without the C++ exe removes a brittle dependency for the most common use case (prompt iteration).

- **The programmatic verification pass (Phase 2c)** is the right level of paranoia — cheap, fast, catches the most common Haiku hallucinations without adding another LLM call.

---

### 3. Concerns & risks (ranked by severity)

**Critical:**

1. **The prompts are the plan's biggest risk and get the least specification.** The plan meticulously describes JSON schemas, CLI flags, and file paths, but the actual system prompts — which determine 80% of output quality — are hand-waved as "new file: `analysis_system.md`" and "new file: `narrative_system.md`." For a plan that will be implemented by a solo developer, the prompts should have at least a detailed outline or first draft. What specific instructions prevent the model from being generic? What Prismata-specific guidance prevents hallucination of game mechanics? The plan describes *what* the prompts contain at a high level but not *how* they're structured.

2. **Few-shot example selection logic (Phase 3a) is fragile.** Selecting examples by game length and upset status is clever but assumes 3 examples will generalize. With only 3 manually-written commentaries, there's no coverage for: rush games (<10 turns), games with unusual win conditions (resignation vs timeout vs breach), games where the AI eval data is unreliable (stepper_reliable = false), or games with units not in any existing example. The plan should explicitly address what happens when no example is a good match.

**High:**

3. **Phase 1b unit knowledge lookup is underspecified.** "Simple text search for unit name headers in the KB files" on a 1,266-line markdown file is more fragile than it sounds. Unit names can appear in multiple sections (synergies, counters, tier lists). Which section do you extract? What's the delimiter? What about units mentioned by nickname or abbreviation? This needs a concrete parsing strategy, not a hand-wave.

4. **The plan conflates "Phase 1 JSON output" with "calling generate_commentary_data.py internally" (Phase 4b).** But `generate_commentary_data.py` is 690 lines with existing CLI behavior. Importing it risks side effects; subprocess means parsing its output or writing temp files. The plan says "subprocess or import" — this needs a decision, not a punt. Given the anti-pattern guard in Phase 4 ("Do NOT import if it has side effects at module level"), it sounds like it *does* have side effects, so subprocess is the answer — but then the `--json-output` flag in Phase 1 is the critical interface and must be rock-solid.

**Medium:**

5. **No handling of edge cases in game data.** What happens with: resigned games (no natural conclusion), games with 0 random set units (base-set only), games where one player times out on turn 3, games where C++ eval shows 50/50 the entire time (no turning points)? The analysis schema requires `turning_points` and `phases` — what if there genuinely aren't any? The prompts need to handle "boring" games gracefully.

6. **Structured output for analysis (Phase 2b) may over-constrain Haiku.** The analysis schema has many required fields (`game_narrative_arc`, `phases`, `turning_points`, `player_assessments`, `set_analysis`, `decisive_factor`, `commentary_hooks`). Structured output mode forces the model to fill every field, which for short/simple games may produce padding or hallucination. Consider making `turning_points` and `commentary_hooks` allow empty arrays, and `set_analysis` optional.

7. **Token budget estimates seem optimistic.** The plan estimates ~4K input for analysis and ~6K input for narrative. But: the condensed KB is ~2,400 tokens, unit knowledge for 8 random set units could easily be ~2K tokens, the game data for a 30-turn game is ~2K tokens, and a single few-shot example is ~3.5K tokens. That puts narrative input alone at ~10K+, not 6K. This affects cost estimates (potentially 50-100% underestimate) and the 8K token guard in Phase 3.

---

### 4. Suggested changes

1. **Move intermediate caching from Phase 4d to Phase 1.** The `--json-output` path should always cache to `bin/commentary/data_{CODE}.json` by default. This is the most expensive step (C++ subprocess) and should never be re-run unnecessarily. Don't wait until Phase 4 to implement this.

2. **Add prompt outlines to Phases 2 and 3.** At minimum, include:
   - The analysis system prompt's section structure (rules summary → unit context → task description → output schema)
   - The narrative system prompt's key instructions (what to emphasize, what to avoid, tone calibration)
   - The narrative user template's variable slots and ordering
   
3. **Decide subprocess vs import for Phase 4b now.** Based on the anti-pattern guard, choose subprocess. Then specify the exact subprocess invocation and error handling (timeout, non-zero exit, invalid JSON output).

4. **Add a `--dry-run` flag early (Phase 2, not Phase 6).** This would run Phase 1 data extraction and print the analysis prompt that *would* be sent to the LLM, without making the API call. Essential for prompt debugging, costs nothing to implement, and saves API dollars during iteration.

5. **Revise token estimates upward.** Recalculate with realistic token counts per component. If narrative input exceeds 8K tokens (likely with few-shot examples), either raise the guard or plan which components to trim first (probably: second few-shot example, then non-notable turn data, then unit knowledge for base-set units).

6. **Phase 2c verification should also check:** Winner identification matches game data, player names aren't swapped, and phase round ranges don't overlap or have gaps (already in checklist but not in the verification pass description).

---

### 5. Alternatives

1. **Single-stage with chain-of-thought instead of two-stage.** For the specific scale (15-turn game, 4-5 messages), a single call with explicit "First, analyze the game in <analysis> tags, then write the commentary" might achieve 85% of the quality at 50% of the cost and complexity. I'd suggest implementing this as a `--single-pass` flag in Phase 2 and empirically comparing against two-stage on the 3 test replays before committing to two-stage as the only path. The research citations support multi-stage for *production systems at scale* — a solo dev passion project may not need it.

2. **Style guide extraction instead of few-shot examples.** Rather than spending ~3.5K tokens per few-shot example, distill the 3 existing commentaries into a ~500-token style guide: sentence length distribution, vocabulary preferences, structural patterns, level of technical depth, how turns are referenced. This would be more token-efficient and more generalizable than hoping the model pattern-matches from 1-2 examples.

3. **Use the C++ eval deltas directly as turning point detection instead of asking the LLM.** The C++ data already identifies the biggest eval swings. Pre-compute turning points in Phase 1 (eval_delta > threshold = turning point) and pass them as structured data. This removes a hallucination vector from Stage 1 and lets the LLM focus on *interpreting* turning points rather than *identifying* them.

---

### 6. Additions

1. **Game length classification and prompt adaptation.** Add a simple classifier in Phase 1: `short` (<12 rounds), `medium` (12-25), `long` (>25). Use this to:
   - Adjust target message count (2-3 for short, 4-6 for medium, 6-8 for long)
   - Select appropriate few-shot examples
   - Modify analysis prompt expectations (short games may have 0-1 turning points)

2. **Graceful degradation specification.** What does the pipeline output when:
   - S3 replay fetch fails (network error, invalid code)?
   - C++ exe crashes or times out?
   - LLM returns malformed analysis JSON despite structured output?
   - LLM narrative doesn't contain `== MESSAGE N ==` delimiters?
   
   Each of these needs a defined behavior. Currently only C++ timeout has a fallback (--eval-only).

3. **A `--replay-json` flag** to accept a local replay JSON file directly, bypassing S3 fetch. Useful for testing with modified/synthetic replays and for offline development.

4. **Token counting before API calls.** Use `anthropic`'s token counting (or a tiktoken estimate) to warn if input exceeds expected budget *before* making the call. This prevents surprise costs on unusually long games.

---

### 7. Removals

1. **Phase 6a A/B comparison tool — remove or defer indefinitely.** For a solo developer with 3 test replays and no automated prose quality metric, a formal A/B comparison tool is over-engineering. Just generate both variants manually and read them. The time spent building comparison tooling would be better spent on prompt iteration.

2. **`--style hype` and `--style casual` (Phase 3d) — remove from initial plan.** The plan acknowledges the audience is "dozens of active readers" who understand Prismata. Building multiple style variants before the default `analytical` style is polished is premature. Add a single `--style` flag with only `analytical` now; the architecture supports adding more later.

3. **Phase 5e cost projection confirmation prompt (`Proceed? [y/N]`)** — remove. For a CLI tool run by the developer who wrote it, interactive confirmation is friction. A `--dry-run` flag that shows costs without submitting is better. If you really want safety, use `--confirm` as an opt-in flag rather than opt-out.

---

### 8. Minor / nits

- The schema in Phase 1a includes `"tier": "rare"` for units, but the context doc doesn't mention a tier system in the replay data. Where does this come from? If it's from the KB, say so explicitly.

- Phase 2b schema uses `"player": int` in turning_points but `"player": str` in player_assessments. Pick one convention (index or name) and use it consistently, or include both.

- The plan references `arXiv 2506.17294` — a paper from June 2025. This is fine but worth noting it's relatively recent and may not be widely validated.

- File naming: `commentary_schema.json` in `tools/` but prompts in `tools/prompts/`. Consider `tools/schemas/` for consistency if more schemas are added later.

- Phase 4c cost tracking shows `$0.006` and `$0.020` which are suspiciously precise for an estimate. Clarify that these are illustrative, not exact.

- The plan mentions preserving "previous commentary versions with timestamp suffix" (Phase 4 checklist) but doesn't specify the format. Suggest: `commentary_FxCfR-K49T+_20260222T1430.txt`.

- Phase 5b says "parallelizable" for Phase 1 data extraction. How? The C++ exe is a single local process. Are you running multiple subprocess instances? This needs a concurrency limit (probably 1, given x86 memory constraints).

---

### 9. Assumptions I'm making

1. **`generate_commentary_data.py` has module-level side effects** (argument parsing, file I/O on import). The Phase 4 anti-pattern guard suggests this, but I haven't seen the code. If it's cleanly structured with `if __name__ == "__main__"`, direct import of its functions would be cleaner than subprocess.

2. **The Anthropic structured output feature works reliably with Haiku 4.5 for the proposed schema complexity.** The schema in Phase 2b has nested arrays of objects with multiple string fields. I'm assuming this doesn't hit edge cases in Haiku's structured output compliance. The plan author should test with a representative schema before building the pipeline around it.

3. **The 3 existing manually-written commentaries are different enough in style/content to serve as effective few-shot examples.** If they're all written by the same person (the solo developer) with the same Claude-assisted workflow, they may all share blind spots.

4. **Prompt caching works within Batch API requests across different games** — i.e., that the shared system prompt prefix is cached for the duration of the batch. The Anthropic docs say batch + caching stack, but I'm assuming the caching actually fires for sequential items in the same batch. The plan author should verify this empirically on a small batch.

5. **The `commentary_prompt.md` (68 lines, ~2,400 tokens) can be meaningfully condensed to ~1,200 tokens** as stated in Phase 2a without losing critical game rules. I have no visibility into this file's content density.

6. **The `docs/commentary-knowledge/03-advanced-units.md` file has a parseable structure** with consistent headers per unit that enables the "simple text search" proposed in Phase 1b. If the file uses inconsistent formatting, this extraction will be brittle.
