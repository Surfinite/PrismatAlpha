# CRITICAL REVIEW OF POST-GAME COMMENTARY PIPELINE PLAN

## 1. One-line verdict
**The plan is overly complex, conflates multiple concerns, and underestimates Haiku's limitations while overengineering around them.**

## 2. What's good (keep as-is)
- **Two-stage LLM pipeline**: Research-backed approach with analysis → narrative separation is sound.
- **Extending existing `generate_commentary_data.py`**: Good leverage of working code, avoids rewriting.
- **JSON schema for structured data**: Excellent for validation and debugging.
- **Batch API integration**: Following proven `discord_knowledge_extractor.py` pattern reduces risk.
- **`--validate-only` mode**: Smart fallback when C++ engine unavailable.
- **Cost tracking**: Essential for solo developer with budget constraints.
- **Few-shot examples from existing commentaries**: Effective for style transfer.

## 3. Concerns & risks (ranked by severity)
**Severity 1 (Critical)**
- **Haiku 4.5 narrative quality likely insufficient**: The manual examples were written with careful prompting and likely Sonnet/Haiku hybrid. Expecting Haiku to match quality for narrative prose is unrealistic. The quality baseline (3 manual commentaries) will not be met without Sonnet.

- **Unit knowledge lookup is naive**: Simple text search for unit names will fail for units with multiple names (e.g., "Tarsier" vs "Tarsiers"), miss context-dependent knowledge, and won't handle synergies between units in the set.

- **No validation of narrative content beyond trivial facts**: Checking turn numbers and unit names catches only simplest hallucinations. LLM can hallucinate strategic assessments ("P1 should have bought X instead"), time pressure effects, or non-existent player traits undetectably.

- **ResourceTracker edge cases unaddressed**: The plan acknowledges <80% reliability for resource validation but proceeds as if validation is ground truth. This will cause factual errors in commentary.

**Severity 2 (Major)**
- **Two-stage pipeline doubles failure modes**: Now both LLM calls can fail independently. Batch processing compounds this—failed analysis stage means narrative stage can't run for any game.

- **Dynamic few-shot selection adds unnecessary complexity**: Selecting examples based on game length/upset status is premature optimization. The token cost difference is minor vs just including the best example.

- **Too many output files**: `analysis_*.json`, `data_*.json`, `commentary_*.txt` creates clutter. Should be single commentary file with analysis appended as metadata.

- **No error recovery strategy**: What happens when C++ `--analyze` times out (300s)? `--eval-only` is mentioned but not integrated into pipeline logic.

- **`--compare` feature over-engineered**: A/B testing is nice but should be Phase 7+, not Phase 6. Quality rubric is more important.

**Severity 3 (Moderate)**
- **Style flags (`--style hype/casual`)**: Creating multiple narrative personas requires separate few-shot examples and prompts. This multiplies development/testing work disproportionately.

- **Programmatic quality rubric is superficial**: Counting factual references doesn't measure commentary quality—just completeness. Human review remains bottleneck.

- **No integration with live commentary system**: Both systems share knowledge base and LLM calls but are developed separately. Opportunity for code reuse missed.

- **Intermediate caching adds state complexity**: Now need to manage `data_*.json`, `analysis_*.json`, track which are stale, etc.

## 5. Alternatives worth considering
**A. Single-stage with structured narrative output**
Instead of free-form narrative, use structured output for both stages:
```json
{
  "messages": [
    {"message_number": 1, "content": "..."},
    {"message_number": 2, "content": "..."}
  ],
  "key_points": ["...", "..."]
}
```
Pros: Guaranteed format, easier validation. Cons: Mechanical prose.

**B. Hybrid single-stage with verification**
Single LLM call generates narrative, second cheap LLM call verifies facts. Less complex than two creative stages.

**C. Extract narrative template from examples**
Instead of few-shot examples, extract rhetorical patterns and create template:
```
Opening: {analysis.opening_summary}
Key moment (turn {tp.round}): {tp.description}
Midgame: {analysis.midgame_summary}
Decisive factor: {analysis.decisive_factor}
Conclusion: {analysis.conclusion}
```
Fill with analysis data. Pros: More control, cheaper. Cons: Less natural.

**D. Use live commentator as base**
The live commentary already has prompt structure and tone. Extend it with full game context rather than building new pipeline.
