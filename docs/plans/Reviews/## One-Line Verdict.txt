## One-Line Verdict

A well-structured, research-grounded plan that extends existing patterns correctly, but has meaningful gaps in robustness: the unit knowledge extraction is fragile, the analysis schema is overambitious for Haiku, the `--validate-only` quality claim is unsupported, and two Phase 3 style flags directly contradict the project's own stated constraints.

---

## What's Good

**Intermediate file caching (Phase 4d).** The three-tier cache (`data_*.json`, `analysis_*.json`, `commentary_*.txt`) is the single best decision in the plan. It decouples the expensive C++ work from prompt iteration, which is exactly the right abstraction for a solo developer tuning prompts. Preserve this exactly as written.

**Anti-pattern guards per phase.** Having explicit "do NOT" rules attached to each phase shows the author understands the failure modes. These will be valuable during implementation. Keep them.

**Dynamic few-shot selection (Phase 3a).** Matching example style to game characteristics (upset → FxCfR, long → uP8mG) is the right call and follows the WSC Sports research cited. The selection criteria are specific enough to implement directly.

**Subprocess boundary decision (implicit).** The note "Do NOT import `generate_commentary_data.py` if it has side effects at module level" is correct instinct. The right call for a 690-line CLI script.

**The verification checklist per phase.** Concrete, actionable, and testable. They read as acceptance criteria, which is what they are.

**Phase ordering and dependency graph.** Correct. No phase starts before its dependencies are complete, and the core deliverable (Phases 1-4) is correctly separated from enhancements (5-6).

**Cost tracking design (Phase 4c).** Printing actual token counts per stage and the batch projection is practical and important for a cost-conscious developer. The format is clear.

## Concerns & Risks

Ranked by severity:

### 1. Unit knowledge extraction is fragile and underspecified (Phase 1b) — HIGH

The plan says "simple text search for unit name headers in the KB files, extract the paragraph(s)." The `03-advanced-units.md` file is 1,266 lines covering 80+ units. If header formats are inconsistent (some `## Plasmafier`, some `### Plasmafier`, some bolded inline), or if unit names are substrings of other unit names (`Drone` appearing inside `Mega Drone` profiles), the extraction will silently return wrong content or nothing. This corrupts the LLM context in a way that's hard to detect — the model will still generate text, just less accurate text.

There's no fallback plan stated when the text search finds nothing. "Fall back to mergedDeck stats" is mentioned but not designed.

**This could cause silent quality degradation on any game containing obscure or multi-word units.**

### 2. Analysis schema is overambitious for Haiku (Phase 2b) — HIGH

The proposed schema requires: game narrative arc, phases with round ranges covering the full game without gaps, turning points with before/after evals, player assessments with strengths/mistakes/notable turns, set analysis, decisive factor, and commentary hooks. That's 8 top-level concepts with nested arrays.

Haiku 4.5 with structured output will fill all fields — it always does — but the content quality of deeply nested reasoning fields is unreliable. Specifically:
- `phases[].rounds` arrays that cover the full game without gaps require the model to reason about boundary conditions it may not handle well
- `player_assessments[].mistakes` will likely be hallucinated for players who played well (the model will fabricate minor errors to have something to say)
- `commentary_hooks` is asking the model to meta-reason about its own output, which is hit-or-miss at Haiku quality

**Risk: analysis stage produces plausible-looking JSON with fabricated content that passes the programmatic verification, then the narrative stage amplifies those fabrications.**

### 3. `--validate-only` mode quality claim is unsupported (Phase 4, Success Criterion #5) — MEDIUM

The context document is explicit: the 3 manually-written commentaries that define the quality bar were all produced using neural eval data. Success criterion #5 says "works without C++ exe" but criterion #3 says "quality parity with manually-written examples." These two criteria cannot both be true for `--validate-only` mode — commentary without eval data will not match commentary with per-turn neural assessment.

The plan should either (a) lower the quality bar for `--validate-only`, or (b) clarify that criterion #3 applies only to the default mode. Currently it's set up to fail on at least one criterion by design.

### 4. Prompt caching in batch mode may not materialize (Phase 5c) — MEDIUM

The plan states the shared knowledge base will be cached across batch items with `"cache_control": {"type": "ephemeral"}`. Ephemeral cache TTL is 5 minutes. Anthropic's Batch API processes asynchronously — there's no guarantee requests for different replay codes are processed within 5 minutes of each other. If a 15-game batch takes 20 minutes to process, most cache reads will miss.

The plan presents caching + batch discount as stacking savings. Caching savings are real for sync mode (where calls happen seconds apart). For batch mode, only the 50% batch discount is reliable. The cost estimates may be optimistic for batch.

### 5. 8K token limit for narrative is soft with no enforcement (Phase 3) — MEDIUM

Phase 3 says "DO NOT exceed 8K input tokens" and "If analysis + examples exceed this, trim the less relevant few-shot example." But:
- Primary example: ~3,500 tokens
- Secondary example (for long games): ~3,500 tokens  
- Analysis JSON: ~2,000 tokens
- System prompt + instructions: ~1,500 tokens
- Per-turn notable turns data: variable

That's already 10,500 tokens before the per-turn data is added. The trimming rule ("drop less relevant example") reduces to ~7,000 tokens, which barely fits. But the plan doesn't define what "trim" means for partial examples, or how to handle a game type that matches all three few-shot selection criteria simultaneously.

**There's no code-level enforcement of the 8K limit. This will silently overshoot and increase costs without the developer noticing.**

### 6. Narrative stage style flags `--style hype` and `--style casual` contradict project constraints (Phase 3d) — MEDIUM

The context document explicitly lists "Commentary for non-expert audiences — out of scope" and "Accessible for non-expert audience" as explicitly excluded. `--style casual` directly contradicts this. `--style hype` is not in the success criteria. Neither style flag has implementation detail (different system prompt? different few-shot examples? just an instruction in the user message?).

These flags should be removed or deferred.

### 7. Programmatic verification misses narrative-level hallucinations (Phase 2c) — LOW-MEDIUM

The verification checks: turn numbers exist, unit names are in deck, eval values match (±2%), player names correct. These are good. But Haiku will also hallucinate:
- "Player X's first Drone buy came on turn 3" when it was turn 2
- "Both players avoided Y" when one player bought Y on turn 8
- Attributing a buy to the wrong player (player index confusion)
- Claiming time pressure on a player who had a full bank

None of these are caught by the current verification plan. The buy sequence is in the JSON and could be cross-referenced.

### 8. `--think-time` flag appears in example but is undefined (Phase 4a) — LOW

```bash
python tools/generate_postgame_commentary.py "FxCfR-K49T+" --think-time 50
```

This flag isn't documented anywhere in the plan. It presumably passes through to the C++ exe, but the plan never specifies what it controls, what the valid range is (context: 50ms default, >200ms causes OOM), or what the default should be.

## Suggested Changes

**Phase 1b — Replace text search with pre-processed unit index.**
Before implementing, create a one-time script `tools/build_unit_knowledge_index.py` that parses `03-advanced-units.md` and outputs `tools/unit_knowledge_index.json` keyed by exact unit name (as it appears in `mergedDeck`). The main pipeline reads from this index. Benefits: reliable at runtime, easy to audit, no regex fragility. The index should be checked into source control. Add a `tools/` README note that the index must be regenerated when the KB is updated.

**Phase 2b — Reduce schema to highest-value fields for V1.**
Drop `phases[].key_decisions` (redundant with `turning_points`), `player_assessments[].strengths` (Haiku will hallucinate these), and `commentary_hooks` (meta-reasoning is unreliable). V1 schema: narrative arc, 2-4 turning points, one player assessment per player (strategy only, mistakes only), decisive factor. This is the minimum viable analysis that the narrative stage actually needs. The fuller schema can be Phase 6 improvements.

**Phase 2b — Add game length adaptation.**
Before constructing the analysis prompt, classify the game:
- Short: < 12 rounds — require only 1-2 turning points, no phase breakdown
- Medium: 12-25 rounds — current schema
- Long: > 25 rounds — add explicit instruction to focus on the 6 highest-swing turns, not all turns

Long games also risk hitting C++ --analyze timeout. The plan should explicitly specify: auto-fall-back to `--eval-only` mode when `total_rounds > 25`.

**Phase 2c — Extend verification to buy attribution.**
Add: for each turning point cited in analysis, verify that the stated buy actually appears in `turns[round].buys` for the stated player. This catches the most common narrative-level hallucination without requiring an LLM.

**Phase 3 — Add token budget calculation as a mandatory step before narrative call.**
Before submitting the narrative API call, calculate input tokens (estimated as character_count / 4) and assert < 7,500 (leaving headroom). If over, drop secondary few-shot example. If still over with single example, log a warning and proceed with no example. This makes the enforcement explicit and automatic.

**Phase 4b — Commit to subprocess, define interface.**
The plan hedges "subprocess or import." Commit to subprocess. Define the exact interface:
```python
result = subprocess.run(
    [sys.executable, "tools/generate_commentary_data.py", code, "--json-output", data_path],
    cwd=project_root, capture_output=True, timeout=360, check=True
)
```
Document that the structured JSON is written to `data_path` (not stdout), which avoids pipe encoding issues.

**Phase 4d — Define the naming convention for versioned files.**
Specify: if `commentary_CODE.txt` exists, rename it to `commentary_CODE_YYYYMMDD_HHMMSS.txt` before writing the new version. This is concrete enough to implement directly and avoids ambiguity.

**Phase 5c — Remove batch + caching savings claim, or qualify it.**
Change the 5c language from "prompt caching active (5-min cache)" to "prompt caching may provide additional savings if batch processes quickly; verify via cache_read_input_tokens in API response." Don't present it as reliable.

## Alternatives

### Alternative 1: Replace analysis LLM stage with a Python data pipeline

The analysis stage's core job is identifying game phases and turning points. The C++ eval already provides eval deltas per turn. Consider:
- Turning points = turns where `|eval_delta| > threshold` (e.g., 8%)
- Phases = inferred from buy composition changes (Drone-heavy → attacker-heavy transition)
- Decisive factor = the highest-swing turn's description

This is deterministic, free, and instantaneous. The LLM is genuinely better at interpreting *why* a turning point mattered, but not at identifying *which* turns are significant — that's just math on the eval data.

**If this worked, it would eliminate one full API call (~$0.006/game savings) and remove Haiku's most reliability-challenged task (reasoning over tabular data).** The narrative stage would receive deterministically correct turning points and could focus purely on prose.

This is worth prototyping on the 3 test replays before committing to the two-stage LLM approach. The eval data might be sufficient to identify the interesting turns automatically.

### Alternative 2: Extended thinking on Sonnet as a single stage

If Claude's extended thinking is available on Sonnet, a single call with a thinking budget could do analysis reasoning internally and then produce polished prose as output — the chain of thought serves as the analysis stage without requiring two API round-trips. Estimated cost: ~$0.04-0.06/game with thinking, but potentially higher quality than two Haiku calls. Worth benchmarking against the two-stage Haiku approach on quality, not just cost.

### Alternative 3: Style control via system prompt injection only

Instead of `--style` flags controlling which few-shot examples are selected, style can be controlled entirely via a single paragraph in the system prompt: "Write in the style of [X]." This is simpler to implement and easier to iterate on. The current approach (different examples per style) multiplies testing complexity by the number of styles.

## Additions

**Missing: error recovery matrix.** What happens if Stage 1 (analysis) succeeds but Stage 2 (narrative) fails? The caching in Phase 4d handles this partially (analysis JSON is saved), but the plan doesn't describe the recovery command. Add: if `analysis_CODE.json` exists and `data_CODE.json` exists, skip to Stage 2. Document this as `--from-analysis` flag or just handle it automatically by checking for cached files.

**Missing: answer to reviewer question #5 (game length handling).** The plan's own External Review Notes list this as a key question but the plan doesn't address it at all. Short rush games (8 turns) will produce nearly empty `phases` arrays and sparse turning points. The commentary for a 8-turn blitz needs a different structure than a 39-turn grind. At minimum, the plan should define what the pipeline does with an 8-turn game.

**Missing: handling buy-sac units in buy validation.** The context explicitly notes "ResourceTracker may miss edge cases around `buySac`." If a turn contains a buy-sac purchase that ResourceTracker gets wrong, the analysis stage will receive incorrect buy data and the verification pass will flag things that look correct but aren't. The plan should document that buy-sac edge cases are a known accuracy limitation and commentary should avoid strong claims about turns where `buys` includes units with sacrifice requirements.

**Missing: output encoding specification.** The context document calls out `cp1252` as a known Windows gotcha. The plan mentions UTF-8 in the Python conventions section but no phase-level anti-pattern guard specifies `open(path, 'w', encoding='utf-8')` for file writes. Add to Phase 4 anti-pattern guards.

**Missing: "golden output" regression baseline.** With no automated prose quality testing, prompt changes can silently degrade quality. Suggest saving a copy of the first good auto-generated commentary for each of the 3 test replays as a reference file. Before shipping prompt changes, manually diff new output against the golden file. This is simple but makes regression visible.

## Removals

**Remove `--style hype` and `--style casual` from Phase 3d.** `--style casual` contradicts the explicit scope exclusion ("Commentary for non-expert audiences — out of scope"). `--style hype` has no corresponding few-shot examples and no implementation detail. Keeping these in the plan adds scope without adding value. The `--style analytical` default is sufficient for V1. Move the others to a "Future Work" appendix if the author wants to preserve the idea.

**Remove the `--compare` flag from Phase 6.** For a solo developer, the intermediate file caching already enables manual comparison (just run twice with different flags, both outputs are saved). The `--compare` tool adds implementation surface for limited benefit. The developer's time is better spent on prompt iteration than diff tooling. If comparison is needed, `diff` on two output files is sufficient.

**Remove `--think-time` from the Phase 4a CLI examples** until it's documented. Unexplained flags in example commands imply they're implemented — if they're not, they'll confuse future-you.

## Minor / Nits

- The plan uses both "Stage 1/2" (for LLM stages) and "Phase 1-6" (for implementation phases). This dual numbering creates minor cognitive load when navigating. Consider renaming LLM stages to "LLM-A" and "LLM-B" or "Analysis Call" and "Narrative Call."

- Phase 2a says "~1,200 tokens, subset of `commentary_prompt.md`" — but `commentary_prompt.md` is 67 lines / ~2,400 tokens. A "subset" of 2,400 tokens that is 1,200 tokens isn't obvious — what's excluded? Define what gets cut.

- Phase 3b references "`commentary-generation-instructions.md` lines 163-178." Hardcoded line numbers in a plan will rot when the file is edited. Replace with section description: "the 'Discord Format Specification' section."

- The JSON schema in Phase 1a uses Python-style type hints (`str`, `[int, int]`) rather than JSON Schema format. Label it clearly as "pseudocode schema" to avoid confusion when creating the actual `commentary_schema.json`.

- Phase 5a batch CLI shows `--batch-codes "FxCfR-K49T+,WjhmP-WWdXx"` with comma separation. But replay codes can theoretically contain commas (the `+` → `_PLUS_` encoding exists for a reason). Consider `--batch-codes` using space separation or a different delimiter, or just requiring a file.

- The research citation `arXiv 2506.17294` has a `2506` prefix — June 2025. Given today's date (Feb 2026), this paper is plausible but the arxiv ID format suggests it was published after the project started. Worth confirming the paper exists and isn't a hallucinated citation. (I'm flagging, not asserting.)

## Assumptions You're Making

*Flag these for the plan author to validate against the actual codebase:*

1. **`03-advanced-units.md` uses consistent markdown headers for unit names.** If units are formatted inconsistently (some as `## Unit Name`, some as bold text `**Unit Name**`, some as HTML), the text search extraction will fail silently. The author should verify the actual header format before designing the extraction.

2. **`generate_commentary_data.py` has no module-level side effects.** The anti-pattern guard warns about this but doesn't confirm it. The author should check whether this file executes any code at import time (file I/O, subprocess calls, etc.). If it does, subprocess is the only safe orchestration path.

3. **`mergedDeck` in the replay JSON contains sufficient unit stats for fallback.** Phase 1b says "fall back to mergedDeck stats if no KB entry found." I'm assuming `mergedDeck` includes readable ability descriptions, not just numeric fields. If it only has cost/HP/build time, the fallback context is thin.

4. **Batch API ephemeral cache TTL behavior with asynchronous processing.** I'm assuming (and flagging as uncertain) that ephemeral caches don't provide meaningful savings for batch mode. The author should test this on a small batch and check `cache_read_input_tokens` in the response to confirm whether caching fires.

5. **The 3 few-shot examples are each roughly ~3,500 tokens.** I estimated this from "45 lines, ~3,500 tokens" for one example. The actual token counts for all three determine whether the 8K narrative budget is workable. The author should measure these with the Anthropic token counter before finalizing Phase 3.

6. **`--analyze` auto-fallback to `--eval-only` for long games isn't already in the existing code.** If `generate_commentary_data.py` already handles this, Phase 1 can skip it. If not, it needs to be added.

7. **The Anthropic structured output feature is stable on Haiku 4.5 for complex nested schemas.** The plan cites "GA on Haiku 4.5" for structured outputs, but the cited schema is significantly more complex than typical structured output examples. Testing with the actual schema on a real replay before finalizing Phase 2 would derisk this.
